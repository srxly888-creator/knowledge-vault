# 企业级最佳实践与优化指南
> 本文把影刀开放 API 的事实约束、仓库内 AI+RPA 架构经验和企业实施经验，压缩成一份生产可执行清单。核验日期：2026-03-24。

## 1. 生产准备度原则
- 先有平台能力，再有业务模板。
- 先有回调/轮询/幂等/账本，再有批量启动。
- 先把 job 与 task 的边界讲清楚，再让业务团队接入。
- 先把安全与审计链路拉通，再把自动化放上主业务链路。

## 2. 凭证与 Token 管理
把 accessKeyId/accessKeySecret 视为平台级密钥，而不是项目级脚本配置。
### 应该做
- 使用专门的 Secret Manager 保存 AK/SK。
- 通过 Token Broker 统一换发 accessToken，减少散点缓存。
- 把 Token 刷新、失效、401 重试逻辑做成公共库。
### 不应该做
- 把 AK/SK 写死在机器人代码、Postman 共享环境或前端页面中。
- 每个服务自己去换 Token，导致风控和缓存不一致。

## 3. 环境与域名管理
影刀公有云、专有云和兼容域名差异必须配置化。
### 应该做
- 把 `api.yingdao.com`、`api.winrobot360.com`、专有云域名都抽成配置项。
- 在 CI/CD 中做环境差异校验，避免误发到生产。
### 不应该做
- 在代码里写死域名。
- 把所有环境都指向同一地址再靠人工切换。

## 4. job 与 task 的选择
先选执行抽象，再写业务逻辑。
### 应该做
- 单应用直连、简单资源选择优先 job。
- 多节点编排、任务模板治理优先 task。
- 把 `jobUuid` 与 `taskUuid` 的生命周期分开管理。
### 不应该做
- 复杂流程强行用 job。
- 把 task 与 job 的 UUID 混用。

## 5. 工作队列优先
批量业务对象不要直接逐个 start，而是优先入列。
### 应该做
- 队列项要带业务主键、优先级、有效期、来源系统、幂等键。
- 按业务域拆分队列，避免一条队列承载所有场景。
### 不应该做
- 高峰时直接并发轰炸 start 接口。
- 把失败对象和正常对象混在同一批重试。

## 6. 参数设计
参数名、类型、必填关系都应与应用主流程保持强一致。
### 应该做
- 建立参数字典与版本号。
- 上线前用真实样本做参数回放测试。
- 对日期、金额、布尔、枚举做显式转换。
### 不应该做
- 依赖自由文本约定参数。
- 让不同应用对同一业务字段使用不同命名。

## 7. 大参数与文件旁路
长度接近阈值时，先文件化再传 file 参数。
### 应该做
- 批量订单、批量发票、长摘要统一走 file/upload。
- 给上传文件设置保留期、脱敏和归档策略。
### 不应该做
- 把大文本直接放在 JSON body 中。
- 上传后的文件缺少生命周期治理。

## 8. 回调优先、轮询兜底
结果获取必须双轨设计。
### 应该做
- 回调收到后先返回 2xx，再异步处理。
- 轮询只扫描未终态对象，并根据运行时长放大间隔。
- 回调与轮询统一写入状态账本。
### 不应该做
- 只依赖轮询。
- 业务异常直接返回 5xx 给回调调用方。

## 9. 终态集合配置化
终态不是文档知识，而是系统配置。
### 应该做
- 把任务、应用、机器人、队列项状态表固化到配置中心。
- 所有轮询器、告警器、报表都复用同一套状态定义。
### 不应该做
- 在不同项目里手写不同终态集合。
- 靠前端展示文案判断流程是否结束。

## 10. 幂等与审计
每次执行都要能追溯。
### 应该做
- start 请求写业务账本；记录 requestId、idempotentUuid、task/job UUID。
- 人工重试、手动停止、审批通过也要留痕。
### 不应该做
- 网络超时后直接重新提交，不校验幂等。
- 日志里看得到错误，账本里却找不到对应业务对象。

## 11. 速率限制治理
限流预算必须成为平台能力。
### 应该做
- 按端点分桶配置，例如日志接口更低。
- 高峰期对日志、列表、搜索类接口进行降频。
### 不应该做
- 只做全局 QPS。
- 把日志搜索当实时主链路调用。

## 12. 机器人池治理
机器人是有限资源，不是无限并发槽位。
### 应该做
- 按业务域、系统权限、网络区域拆分机器人组。
- 监控 idle、offline、allocated、running 比例。
- 热组与冷组分开，避免热点拥塞。
### 不应该做
- 财务与营销共用同一组机器人。
- 不监控离线率和排队时长。

## 13. 日志与截图治理
运行日志和截图要进入统一可观测平台。
### 应该做
- 把 job/log/search 与截图 URL 收敛到异常中心。
- 日志保留原文，同时产出 AI 摘要与根因标签。
### 不应该做
- 只在影刀控制台里看日志，不做平台归档。
- 截图散落在对象存储，无法和业务单据关联。

## 14. 测试体系
生产前必须做接口、参数、模板、机器人、回调五层测试。
### 应该做
- 接口层：鉴权、域名、限流、错误码。
- 参数层：必填校验、边界值、文件参数。
- 模板层：应用版本、任务模板、回调配置。
- 资源层：机器人登录态、网络、分组路由。
- 结果层：回调、轮询、账本一致性。
### 不应该做
- 只在开发机上手点成功一次就上线。
- 没有失败样本与回滚演练。

## 15. 版本与变更
把应用、任务模板、Prompt、策略都纳入版本治理。
### 应该做
- 应用发版前先做灰度。
- 重大模板变更要保留旧版本回滚入口。
- Prompt、参数映射、审批规则都要可追踪。
### 不应该做
- 修改模板即覆盖生产。
- 无法回答昨天和今天为什么执行结果不同。

## 16. AI 治理
AI 负责增强，而不是绕过治理。
### 应该做
- 高风险动作必须审批。
- 结构化输出必须经过校验再执行。
- AI 计划、解释、摘要都保留版本与置信度。
### 不应该做
- 让模型自由拼装接口请求体直接发执行。
- 没有置信度和审批阈值。

## 17. 运维 Runbook
常见问题必须标准化处理。
### 应该做
- 401：检查 AK/SK、管理员账号、Token Broker。
- 429：看端点预算、批次大小、日志扫描频率。
- 回调缺失：看网关、白名单、2xx ack 逻辑。
- 机器人离线：看客户端、远程桌面、资源。
### 不应该做
- 每次故障都从头排查。
- 故障只在聊天记录里，没有沉淀到 Runbook。

## 18. 生产上线检查清单
- [ ] 域名、证书、公有云/专有云地址是否都已配置化？
- [ ] AK/SK 是否存放于 Secret Manager，且有轮换流程？
- [ ] 是否实现了 Token 缓存与 401 自动刷新？
- [ ] 是否为 start 请求记录幂等键和状态账本？
- [ ] 是否已经打通回调连通性、2xx ack 和轮询兜底？
- [ ] 是否完成参数边界值测试和 file/upload 旁路测试？
- [ ] 是否建立机器人状态与容量监控？
- [ ] 是否按端点完成限流预算？
- [ ] 是否有错误分类、告警路由和值班 Runbook？
- [ ] 是否对高风险动作启用了审批？

## 19. 常见优化方向
| 方向 | 做法 | 预期效果 |
| --- | --- | --- |
| 减少等待 | 优先选 idle 机器人、错峰轮询、对高峰对象先入队列 | 降低排队时长 |
| 减少失败 | 参数字典化、FAQ 归因、失败样本回放 | 降低参数错误率 |
| 减少人工 | AI 摘要日志、自动分类异常、批量文件旁路 | 缩短排障时间 |
| 提升审计 | 账本化、截图归档、requestId 关联 | 提升追溯性 |
| 降低成本 | 拆分模板、复用机器人组、减少无效轮询和日志扫描 | 提升单位资源产出 |

## 20. 结语
影刀企业级落地的难点从来不只是“会不会调接口”，而是“能不能持续稳定地运营自动化能力”。最佳实践的意义，就是把文档里的零散事实转成平台里的固定机制。
